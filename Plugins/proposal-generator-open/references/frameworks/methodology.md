# Strategic Methodology Reference

## Overview

This document contains the strategic frameworks that should be applied throughout the proposal process. These frameworks ensure proposals are:
- Strategically grounded
- Consistently structured
- Methodologically sound

---

## The 5-Phase Strategic Framework

### Purpose
Ensures strategic thinking BEFORE implementation. Prevents jumping to solutions without understanding problems.

### The Five Phases

```
T → H → I → N → K
│   │   │   │   │
│   │   │   │   └── KNOWLEDGE: Document for replication
│   │   │   └────── NETWORK: Identify expansion opportunities
│   │   └────────── INVEST: Evaluate and decide
│   └────────────── HYPOTHESIS: Propose solutions
└────────────────── TASK: Identify the manual work
```

### Phase 1: TASK
**Identify what manual work is actually happening**

Key Questions:
- What manual process is slowing growth?
- How many hours per week does this consume?
- What's the cost of NOT automating this?
- What's breaking at current scale?
- Who's doing the work? (role, seniority, cost)

Output:
```
TASK ANALYSIS
Manual Task: [SPECIFIC TASK]

Impact Metrics:
- Hours/week: [X]
- People involved: [ROLE, SENIORITY]
- Cost: [$/week or $/transaction]
- Opportunity cost: [WHAT'S NOT HAPPENING]

Priority: [CRITICAL/HIGH/MEDIUM]
```

### Phase 2: HYPOTHESIS
**Propose what solutions could look like**

Key Questions:
- If fully automated, what would change?
- What's the success metric?
- What data/systems would need to connect?
- What are 3+ different approaches?
- What could go wrong with each?

Output:
```
HYPOTHESIS ANALYSIS
Task Being Automated: [FROM TASK PHASE]

Option 1: [SPECIFIC APPROACH]
- Success Metric: [MEASURABLE OUTCOME]
- Trade-offs: [WHAT YOU GAIN/LOSE]
- Risk: [WHAT COULD FAIL]

Option 2: [DIFFERENT APPROACH]
...

Recommended Path: [WHY THIS OPTION?]
```

### Phase 3: INVEST
**Evaluate options and make investment decision**

Three-Phase Logic:
1. **Filter** - Eliminate options that fail basic criteria
2. **Score** - Rate viable options on key dimensions
3. **Decide** - Select option with best score + ROI

Output:
```
INVESTMENT DECISION
Recommended Option: [OPTION NAME]

Scoring:
- Problem Fit: [X/10]
- Implementation Feasibility: [X/10]
- ROI Potential: [X/10]
- Risk Level: [LOW/MEDIUM/HIGH]

Investment Required: $[AMOUNT]
Expected Return: $[AMOUNT]
Break-even: [TIMEFRAME]
```

### Phase 4: NETWORK
**Identify other applications and stakeholders**

Key Questions:
- Who else has a similar problem?
- What other use cases fit this pattern?
- How does this connect to other systems?
- What's the scaling potential?

### Phase 5: KNOWLEDGE
**Document for replication**

Output:
```
KNOWLEDGE DOCUMENTATION
Problem Template: [INDUSTRY/PROBLEM TYPE]
Solution Pattern: [PROVEN APPROACH]
Implementation Playbook: [STEP-BY-STEP]
Success Metrics: [WHAT TO MEASURE]
Lessons Learned: [WHAT WORKED, WHAT DIDN'T]
```

## When to Apply the Framework

### In Proposals

| Proposal Section | Framework Phase to Reference |
|------------------|------------------------------|
| Understanding of Needs | TASK phase analysis |
| Proposed Solution | HYPOTHESIS phase options |
| Why This Approach | INVEST phase scoring |
| Future Opportunities | NETWORK phase expansion |
| Methodology Section | Full framework overview |

### Framework Language in Proposals

DO use:
- "Our first step is understanding the actual manual work..."
- "We evaluated three approaches and recommend..."
- "This solution scored highest on implementation feasibility..."

DON'T use:
- Technical framework names in executive summaries
- Jargon the client won't understand
- Framework-heavy language in commercial proposals

## Scoring Framework (6 Dimensions)

### Purpose
Provides objective, repeatable scoring for comparing options.

### The Six Dimensions

| Dimension | What It Measures | Key Question |
|-----------|------------------|--------------|
| **S**egment | User routing capability | Can we route by user type? |
| **P**roblem | Problem-solving fit | Does it solve the actual problem? |
| **E**valuate | Resource optimization | Is it efficient? |
| **E**xecute | Implementation feasibility | Can we actually build this? |
| **D**isplay | Insight presentation | Does it communicate well? |
| **S**et | Continuous improvement | Can it optimize over time? |

### Scoring Scale

| Score | Meaning |
|-------|---------|
| 9-10 | Exceptional - exceeds requirements |
| 7-8 | Strong - meets all requirements |
| 5-6 | Adequate - meets minimum requirements |
| 3-4 | Weak - significant gaps |
| 1-2 | Failing - does not meet requirements |

### Scorecard Output

```
OPTION SCORECARD

Option: [NAME]

| Dimension | Score | Rationale |
|-----------|-------|-----------|
| Segment   | [X/10] | [WHY] |
| Problem   | [X/10] | [WHY] |
| Evaluate  | [X/10] | [WHY] |
| Execute   | [X/10] | [WHY] |
| Display   | [X/10] | [WHY] |
| Set       | [X/10] | [WHY] |
| TOTAL     | [X.X/10] | |

Status: [STRONG/ADEQUATE/CONCERN]
Recommendation: [PROCEED/CONDITIONAL/DO NOT PROCEED]
```

### When to Include Scorecard

| Deal Type | Include Scorecard? |
|-----------|-------------------|
| GOVERNMENT | Yes - in Appendix |
| COMMERCIAL | Optional - reference in text |
| PARTNERSHIP | No - too formal |
| SIMPLE/SOW | No - overkill |

## Human Strategic Layer

### Purpose
Ensures human judgment overlays any framework scoring. Frameworks inform, humans decide.

### Key Questions

After running any framework analysis, ask:

1. **Synthesis**: What's the ONE most important thing here?
2. **Context**: What business reality isn't captured in the scores?
3. **Trade-offs**: What are we gaining and losing with this approach?
4. **Risk**: What could go wrong that the framework didn't catch?
5. **Gut Check**: Does this feel right given what we know?

### Application in Proposals

The human layer should be evident in:
- Executive Summary - synthesized insight, not just data
- Positioning - strategic choices explained
- Risk Section - real risks, not boilerplate
- Next Steps - clear, actionable, specific

## Quality Markers

A good proposal using this methodology will:

1. **Show Understanding** - TASK phase analysis is evident
2. **Present Options** - HYPOTHESIS phase thinking is clear
3. **Justify Recommendation** - INVEST phase logic is sound
4. **See Beyond** - NETWORK phase opportunity is visible
5. **Be Repeatable** - KNOWLEDGE phase documentation exists

---

## Common Mistakes

| Mistake | Why It's a Problem | Fix |
|---------|-------------------|-----|
| Jumping to solution | Skips TASK phase | Always start with problem analysis |
| Single option | Skips HYPOTHESIS phase | Present at least 2-3 options |
| No scoring | Skips INVEST phase | Show decision logic |
| Standalone project | Skips NETWORK phase | Connect to bigger picture |
| No documentation | Skips KNOWLEDGE phase | Include methodology handoff |
